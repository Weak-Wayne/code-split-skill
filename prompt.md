## 核心规则

### 规则一：禁止单文件模式
**绝不允许**将所有业务逻辑写在一个文件中。

### 规则二：强制模块分析
在编写任何代码之前，**必须先输出模块功能分析文档**。

### 规则三：模块分类标准

| 模块类型 | 目录 | 职责说明 |
|---------|------|----------|
| 认证模块 | `auth/` | 用户登录、注册、Token管理、权限验证 |
| API模块 | `api/` | 后端接口调用、请求封装、响应处理、Mock数据 |
| 组件模块 | `components/` | UI组件、页面组件、可复用组件、布局组件 |
| 工具函数 | `utils/` | 通用工具函数、格式化函数、验证函数、辅助函数 |
| 自定义Hooks | `hooks/` | 业务逻辑hooks、状态管理hooks、副作用hooks |
| 配置文件 | `config/` | 环境变量配置、全局配置、常量定义 |
| 类型定义 | `types/` | TypeScript类型定义、接口声明、枚举定义 |
| 状态管理 | `stores/` | 全局状态管理（如Zustand、Pinia） |
| 布局组件 | `layouts/` | 页面布局组件 |
| 页面组件 | `pages/` | 页面级组件 |

### 规则四：文件命名规范
- 组件文件使用大驼峰：`UserProfile.tsx`
- 工具函数使用小驼峰或连字符：`formatDate.ts`, `string-utils.ts`
- 类型定义文件使用小写：`user.ts`
- 接口使用大驼峰：`interface User {}`

### 规则五：代码行数限制
- 单个文件不超过200行
- 单个函数不超过50行

## 执行流程

### 第一步：模块功能分析（必须输出）
在编写代码前，必须输出模块分析文档：

```markdown
## 模块功能分析

### 1. [模块名称] ([目录]/)
- 职责：[职责描述]
- 文件：`src/[目录]/[文件1].ts`, `src/[目录]/[文件2].ts`

### 2. [模块名称] ([目录]/)
- 职责：[职责描述]
- 文件：`src/[目录]/[文件1].ts`

### 文件结构预览
```
src/
├── [目录1]/
│   └── [文件1].ts
├── [目录2]/
│   └── [文件1].ts
└── index.ts
```
```

### 第二步：按模块编写代码
严格按照分析文档创建文件，每个文件只包含单一职责的代码。

### 第三步：统一导出
通过 `index.ts` 文件统一导出模块：

```typescript
// src/auth/index.ts
export { login } from './login';
export { register } from './register';
```

## 错误示例（禁止）

```
// ❌ 错误：所有代码在一个文件中
app.js
├── 登录逻辑
├── 用户信息展示
├── API调用
├── 工具函数
└── 配置常量
```

## 正确示例（推荐）

```
// ✅ 正确：按模块拆分
src/
├── auth/
│   └── login.ts          # 登录逻辑
├── api/
│   └── user-api.ts       # 用户API
├── components/
│   └── UserProfile.tsx   # 用户组件
├── utils/
│   └── format.ts         # 工具函数
└── config/
    └── constants.ts      # 配置常量
```

## 强制要求

- ✅ **必须**在编写代码前输出模块分析文档
- ✅ **必须**遵循单一职责原则
- ✅ **必须**使用index.ts进行模块导出
- ✅ **必须**保持文件独立性，避免循环依赖
- ✅ **必须**遵守文件行数限制（≤200行/文件）
- ❌ **禁止**在一个文件中混合多个模块类型
- ❌ **禁止**将所有代码写在单个文件中
